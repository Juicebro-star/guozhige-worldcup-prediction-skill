# 赛果期望值与方案优化模型

本文件用于"方案研究/玩法选择/几串几/组合优化"场景。定位是模型研究和赛后复盘,不是投注建议。

---

## 1. 最高原则

不要问:

```text
哪个最稳?
哪个奖金最高?
```

要问:

```text
哪个结果的模型概率高于市场隐含概率?
哪个玩法最贴合这场比赛的信息结构?
哪个组合在期望、命中概率、波动之间最均衡?
```

对外要翻译成:

```text
哪个方向市场看轻了?
这场到底适合看胜平负,还是看进球数?
几场组合最顺,别硬凑高奖金?
```

口径:

- 可以说"研究价值高/低""纳入研究池/不纳入研究池""适合赛后复盘"。
- 不说"买""上""冲""稳胆""下注金额""推荐投注"。
- 能说"划算/不划算""给得够不够香""稳但不值""有研究味儿"。
- 除非用户明确要看公式,否则最终回答少用英文缩写,少堆复杂表格。

## 2. 每次必须抓取的实时数据

方案研究比普通赛前判断更依赖最新数据。每次执行前必须查:

1. 最新赛程和比赛时间,统一到用户所在时区;
2. 最新赛果和晋级路径;
3. 伤停、停赛、红黄牌、预计首发;
4. 场地、天气、旅行和休息天数;
5. 各玩法当前回报倍数/赔率;
6. 单关/过关/混合过关是否可用;
7. 赔率更新时间或来源时间。

若 Sporttery 官方数据接口可用,优先使用官方计算器/官方接口;若不可用,用 ESPN、官方赛事页、Oddschecker、Covers、主流数据站交叉验证,并在输出里说明"本次用公开市场数据替代官方回报倍数"。

## 3. 统一事件模型

把所有玩法统一为事件:

```text
event = match + pool + outcome
```

示例:

```text
阿根廷 vs 佛得角 + 胜平负 + 阿根廷胜
阿根廷 vs 佛得角 + 总进球数 + 2球
澳大利亚 vs 埃及 + 半全场 + 平平
哥伦比亚 vs 加纳 + 比分 + 2-1
```

每个事件记录:

| 字段 | 含义 |
|---|---|
| `p_model` | 模型估计真实概率 |
| `odds` | 当前十进制回报倍数 |
| `p_market` | 市场隐含概率,`1 / odds` |
| `edge` | 概率差,`p_model - p_market` |
| `ev` | 理论期望值,`p_model * odds - 1` |
| `confidence` | 置信度,高/中/低 |
| `fit` | 玩法适配度,高/中/低 |
| `risk_note` | 主要风险 |

对外表格可以改成 100% 鲜榨果汁味列名:

| 内部字段 | 对外列名 |
|---|---|
| `p_model` | 果汁哥估计 |
| `p_market` | 市场门槛 |
| `edge/ev` | 划不划算 |
| `fit` | 适不适合这玩法 |
| `risk_note` | 坑在哪 |

## 4. 从比分分布映射玩法

优先先建比分概率分布,再映射到各玩法。不要给胜平负、总进球数、比分各算一套互相打架的概率。

示例:

```text
0-0: 8%
1-0: 13%
1-1: 15%
2-0: 12%
2-1: 14%
0-1: 9%
1-2: 8%
其它: 21%
```

映射:

```text
胜平负主胜概率 = 所有主胜比分概率求和
平局概率 = 所有平局比分概率求和
客胜概率 = 所有客胜比分概率求和
总进球2球概率 = 2-0 + 1-1 + 0-2
比分2-1概率 = 单个2-1概率
```

半全场需要额外判断进球时间和上半场结构,不能只从全场比分硬推。

## 5. 单事件评分

基础公式:

```text
p_market = 1 / odds
edge = p_model - p_market
ev = p_model * odds - 1
```

综合值:

```text
score = ev * confidence_weight * fit_weight - risk_penalty
```

建议权重:

| 项 | 高 | 中 | 低 |
|---|---:|---:|---:|
| `confidence_weight` | 1.00 | 0.75 | 0.50 |
| `fit_weight` | 1.00 | 0.80 | 0.55 |

风险惩罚:

| 风险 | 惩罚 |
|---|---:|
| 赔率/数据来源不是官方最新 | 0.03 |
| 关键伤停未确认 | 0.04 |
| 半全场剧本不清楚 | 0.06 |
| 比分事件分布太散 | 0.05 |
| 热门低回报且 edge 接近 0 | 0.04 |

纳入研究池的基本门槛:

```text
ev > 0
edge >= 0.03
confidence != 低
fit != 低
```

低回报热门要更严格:

```text
odds < 1.50 时,edge 至少 >= 0.06
```

输出时不要把这套公式全倒出来。默认只讲:

```text
果汁哥估计它机会更大,市场给的门槛没那么高,所以有研究味儿。
果汁哥也看好它能中,但市场已经把它捧太热了,所以稳但不值。
```

## 6. 玩法选择规则

每场方案研究必须做"玩法矩阵",至少扫一遍:

```text
胜平负:方向是否清楚,回报是否被压扁?
让球胜平负:强队赢几个/弱队能不能守住门槛?
总进球数:比分区间是否集中?
比分:主比分和备选比分是否足够集中?
半全场:上半场剧本是否清楚?
```

如果缺少某玩法实时数字,输出"待数字确认",但仍给结构判断。

### 胜平负

适合:

- 主方向清楚;
- 市场没有把回报压得太低;
- 对手缺少进球或拖平能力;
- 模型胜率明显高于盈亏平衡概率。

常见陷阱:

```text
强队胜率高,但回报太低,数学期望不够。
```

100%鲜榨果汁味:

```text
这队大概率能赢,但大家都知道它能赢,所以给得不香。
```

### 总进球数

适合:

- 主比分和备选比分落在同一进球区间;
- 胜平负方向有分歧,但节奏/总球数更清楚;
- 强队可能赢,但赢法更像 2-0 / 2-1 / 3-0 这种集中分布。

100%鲜榨果汁味:

```text
总进球像估一顿饭几道菜,不用猜每道菜名,但你得知道这桌大概是两三道,还是满汉全席。
```

### 比分

适合:

- 比分分布高度集中;
- 某个比分概率明显高于市场隐含概率;
- "其它比分"风险可控。

一般只纳入激进研究,不作为稳健主方案。

100%鲜榨果汁味:

```text
比分这玩意儿太细,说中了很爽,但别把它当主饭吃。
```

### 半全场

适合:

- 强队开局压制非常强:胜/胜;
- 强队慢热但后程强:平/胜;
- 双方极度谨慎:平/平。

若进球时间不清楚,不要硬做半全场。

100%鲜榨果汁味:

```text
半全场不是看强弱,是看剧本。剧本不清楚,别硬演。
```

### 混合过关

适合:

- 每个事件都单独正 EV;
- 同场事件相关性能解释,且不是重复押同一个叙事;
- 串关后组合 EV 仍为正;
- 命中概率没有低到只剩彩票化。

## 7. 组合与串关模型

独立事件组合:

```text
P_combo = p1 * p2 * ... * pn
O_combo = o1 * o2 * ... * on
EV_combo = P_combo * O_combo - 1
```

但足球事件经常相关。处理规则:

1. 不同比赛可以近似独立;
2. 同一场比赛内的事件必须从比分分布联合计算;
3. 同一叙事的事件不能重复当作独立优势;
4. 组合越长,风险惩罚越大。

串关风险惩罚:

| 方式 | 风险 |
|---|---|
| 单关 | 低 |
| 2串1 | 中低 |
| 3串1 | 中高 |
| 4串1及以上 | 高,默认只做激进样本 |
| 多个比分/半全场串关 | 极高 |

默认排序:

```text
单关正EV > 2串1正EV > 3串1正EV > 高串
```

除非高串的每一项都有极高 edge 和清晰独立性,否则不作为数学最优。

100%鲜榨果汁味:

```text
能两场说清楚,就别硬凑三场。串得越长,容错越低。
最高奖金不是最优,最能解释清楚的方案才有复盘价值。
```

---

## 8. 输出分层

每次方案研究必须输出四层:

1. **最有研究味儿**: 综合最划算;
2. **最稳妥的研究方向**: 回报不一定最高,但逻辑最顺;
3. **激进看戏方案**: 回报高、波动大,只用于复盘观察;
4. **先别碰的方向**: 说明为什么放弃。

输出口径示例:

```text
兄弟们,这不是让你照着上,这是果汁哥做模型复盘。数学上最干净的是总进球2球,不是强队胜。强队胜可能中,但市场也知道它强,回报被压扁了,这就叫稳但不值。
```

---

## 9. 赛后复盘

复盘不要只看中没中,要看四件事:

1. `p_model` 是否高估/低估;
2. 市场隐含概率是否更接近真实;
3. 玩法选择是否贴合比赛结构;
4. 串关失败是单场判断错,还是组合相关性/方差问题。

| 事件 | 模型概率 | 市场隐含概率 | 结果 | 复盘 |
|---|---:|---:|---|---|
| {event} | {p_model} | {p_market} | 中/未中 | {原因} |
